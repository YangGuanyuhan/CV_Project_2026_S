\maketitle

\begin{abstract}
Open-vocabulary object detection extends traditional detection to arbitrary text-described categories, enabling models to detect objects beyond a fixed training vocabulary. This report presents the reproduction of Grounding DINO, a state-of-the-art open-vocabulary detector, and its zero-shot evaluation on the COCO 2017 validation set. Using the official Swin-Tiny backbone checkpoint, we achieve an AP of 0.4055 (AP50=0.5317) on the full val2017 split. We further conduct ablation studies on detection threshold sensitivity and prompt format design, revealing that the dot-separated prompt format is essential for performance and that lowering the box threshold from 0.35 to 0.25 improves AP from 0.4382 to 0.4637 on a 500-image subset. Qualitative analysis of success and failure cases identifies small-object detection and crowded-scene confusion as primary limitations. All experiments use the official \texttt{groundingdino-py} inference backend without fine-tuning, demonstrating the model's zero-shot capabilities.
\end{abstract}

\begin{IEEEkeywords}
Open-vocabulary object detection, Grounding DINO, COCO, zero-shot detection, vision-language models
\end{IEEEkeywords}

\section{Introduction}
% ============================================================

Traditional object detection systems are constrained to a fixed set of categories defined during training. Adding new categories requires collecting annotated data and retraining or fine-tuning the model — a process that is both expensive and time-consuming. Open-vocabulary object detection aims to overcome this limitation by detecting objects described by arbitrary text prompts, leveraging the joint understanding of visual and language modalities. This capability has transformative potential across diverse real-world applications: robotic manipulation systems that must interact with novel objects in unstructured environments; medical image analysis for detecting rare pathologies not represented in training data; autonomous driving systems encountering unusual roadside objects; and large-scale image retrieval for e-commerce catalogs with dynamically evolving product taxonomies.

Formally, given an input image $I \in \mathbb{R}^{H \times W \times 3}$ and a text query $T$ describing a set of categories $C_T = \{c_1, c_2, \ldots, c_K\}$, an open-vocabulary detector must produce a set of $N$ detections $\{(b_i, c_i, s_i)\}_{i=1}^N$, where $b_i = (x_i, y_i, w_i, h_i)$ denotes a bounding box in COCO coordinate format, $c_i \in C_T$ is the predicted category label, and $s_i \in [0,1]$ is the detection confidence score. Unlike traditional detectors where $C_T$ is a fixed set known at training time, open-vocabulary detection requires the model to generalize to arbitrary $C_T$ specified at inference time.

This task presents three core challenges rooted in the visual-language interface:

\begin{itemize}
    \item \textbf{Visual-language alignment}: The model must understand both image content and text descriptions, mapping semantic concepts to visual regions. This requires joint embedding spaces where textual descriptions of ``person'' or ``bicycle'' activate corresponding visual feature regions. Contrastive pre-training paradigms (e.g., CLIP~\cite{Radford21}) and cross-modal fusion architectures (e.g., Grounding DINO) represent two dominant approaches to this challenge.
    \item \textbf{Generalization}: The model must detect categories unseen during training, requiring robust visual-language representations that do not overfit to training category distributions. Distribution shift between training domains (e.g., Objects365, OpenImages) and test domains (e.g., COCO) tests the model's ability to transfer visual concepts across data sources.
    \item \textbf{Fine-grained grounding}: The model must localize specific objects described by natural language phrases with precise bounding boxes. This demands token-level alignment between text phrases and image regions — the phrase ``red car on the left'' must activate only the corresponding object, not all cars in the scene.
\end{itemize}

This project reproduces the Grounding DINO~\cite{Liu23} zero-shot object detection pipeline and evaluates it on the COCO 2017 validation set. Our specific contributions are: (1) a complete zero-shot evaluation on the full COCO val2017 benchmark (5,000 images) achieving $\text{AP}=0.4055$ ($\text{AP}_{50}=0.5317$) using the Swin-Tiny backbone; (2) threshold sensitivity ablation experiments revealing that the optimal operating point ($\text{box\_threshold}=0.25$) improves subset AP from 0.4382 to 0.4637; (3) prompt format comparison experiments quantifying the critical importance of dot-separated category enumeration, with alternative formats causing approximately 62\% AP degradation; (4) a structured error taxonomy identifying small-object detection, crowded-scene confusion, and occlusion as the primary failure modes; and (5) an open-source, config-driven reproduction pipeline built on the official \texttt{groundingdino-py} package. Training from scratch and fine-tuning on COCO are outside the scope of this reproduction.

The remainder of this report is organized as follows. Section~2 reviews related work on Grounding DINO, open-vocabulary detection, and the COCO benchmark. Section~3 describes our reproduction methodology, including model configuration, inference pipeline, and prompt design. Section~4 presents experimental results across four dimensions: main evaluation, threshold ablation, prompt format ablation, and qualitative analysis. Section~5 discusses limitations, compares our results with reported benchmarks, and outlines future work directions. Section~6 concludes with key findings.

\section{Related Works}
% ============================================================

\subsection{DETR and DINO: Foundations}

Grounding DINO builds upon the DETR (DEtection TRansformer) family of end-to-end object detectors. DETR~\cite{Carion20} reformulated object detection as a set prediction problem using a Transformer encoder-decoder architecture with learnable object queries, eliminating the need for hand-crafted components such as anchor boxes and non-maximum suppression. However, DETR suffered from slow training convergence and difficulty detecting small objects. Deformable DETR~\cite{Zhu21} addressed these limitations by introducing deformable attention modules that attend to sparse spatial locations around reference points, achieving faster convergence and improved multi-scale feature handling.

DINO~\cite{Zhang23} (DETR with Improved denoiSing anchOr boxes) further advanced the DETR paradigm through three key innovations: contrastive denoising training, which accelerates convergence by training the model to reconstruct ground-truth objects from noised queries; mixed query selection, which initializes object queries from encoder features rather than learned embeddings; and a look-forward-twice scheme for box prediction refinement. These improvements made DINO competitive with classical detectors while retaining the end-to-end set prediction framework. Grounding DINO~\cite{Liu23} extends DINO by fusing text features from a BERT language encoder into the DINO detection architecture, enabling the object queries to become text-conditioned for open-vocabulary detection.

\subsection{Grounding DINO}

Grounding DINO~\cite{Liu23} extends the DINO detection architecture with grounded pre-training for open-set detection. The key architectural components include:

\begin{itemize}
    \item \textbf{Swin Transformer backbone}~\cite{Liu21}: Provides hierarchical visual features through four stages with decreasing spatial resolution ($H/4 \times W/4$ to $H/32 \times W/32$) and increasing channel dimension (96 to 768). The shifted-window attention mechanism enables cross-window information exchange while maintaining linear computational complexity with respect to image size.
    \item \textbf{BERT language encoder}~\cite{Devlin19}: Encodes free-form text prompts into contextualized token embeddings. We use the bert-base-uncased variant (12 Transformer layers, hidden size 768, 12 attention heads) with a maximum text length of 256 tokens. The bidirectional self-attention in BERT captures both left and right context for each token, producing phrase-aware representations essential for category disambiguation.
    \item \textbf{Deformable attention fusion}: Six cross-modal feature fusion layers, each with 8 attention heads and 256-dimensional features, align visual regions with text phrases through deformable attention. Unlike standard cross-attention, deformable attention samples sparse spatial locations around learned reference points, reducing computational cost from $O(HW \cdot L)$ to $O(K \cdot L)$ where $K$ is the number of sampling points and $L$ is the text length.
\end{itemize}

The model uses 900 object queries that interact with both visual features from the Swin-T backbone and text features from BERT through the fusion layers. Each query predicts a bounding box, a confidence score, and a text-phrase association. The model is pre-trained on diverse detection and grounding datasets including Objects365 ($\sim$2M images, 365 categories), OpenImages ($\sim$1.7M images, $\sim$600 categories), and GoldG (referring expression grounding data), enabling zero-shot transfer to unseen categories without fine-tuning.

\subsection{Backbone Architectures}

\textbf{Swin Transformer}~\cite{Liu21}. Unlike Vision Transformers (ViT) that process images as a flat sequence of patches with global self-attention, Swin Transformer constructs hierarchical feature maps by merging patches in deeper layers, analogous to the pyramidal feature hierarchy in convolutional networks. The key component is shifted-window multi-head self-attention (SW-MSA), which computes attention within local windows and alternates window partitioning between consecutive layers to enable cross-window connections. The Swin-Tiny variant used in this project has a window size of 7, embedding dimension of 96, stage depths of $[2,2,6,2]$, and attention head counts of $[3,6,12,24]$, totaling approximately 28 million parameters.

\textbf{BERT}~\cite{Devlin19}. Bidirectional Encoder Representations from Transformers (BERT) provides deep contextualized word representations through masked language modeling and next-sentence prediction pre-training. For open-vocabulary detection, BERT's key advantage is its ability to produce context-dependent token embeddings: the word ``bank'' receives different representations in ``river bank'' versus ``bank building,'' enabling the model to resolve semantic ambiguity when mapping detected phrases to COCO categories. The bert-base-uncased model used in Grounding DINO converts input text into WordPiece tokens with a 30,522-token vocabulary, producing a $(L, 768)$ feature matrix for a text of length $L$.

    \item \textbf{GLIP}~\cite{Li22}: Grounded language-image pre-training formulates object detection as a phrase grounding problem, unifying detection and visual grounding through a single architecture. Among these, Grounding DINO is chosen for our reproduction due to its strong zero-shot performance, well-maintained open-source implementation (\texttt{groundingdino-py}), and the architectural clarity of its DINO-based design which separates visual encoding, language encoding, and cross-modal fusion into distinct, analyzable components.

\subsection{COCO Benchmark}

Microsoft COCO (Common Objects in Context)~\cite{Lin14} is the standard benchmark for object detection evaluation. The dataset contains 80 object categories spanning common everyday objects, with a characteristic long-tail category distribution: frequent categories such as ``person'' ($\sim$262K training instances) coexist with rare categories such as ``hair drier'' ($\sim$100 instances). The 2017 validation set contains 5,000 images with 36,781 ground-truth bounding box annotations in COCO's $[x, y, w, h]$ format. Evaluation uses the standard COCO metrics computed by the official pycocotools toolkit: average precision (AP) averaged over 10 IoU thresholds from 0.50 to 0.95 with step size 0.05, along with per-size breakdowns (APS: area $<32^2$, APM: $32^2$--$96^2$, APL: $>96^2$) and average recall (AR) at 1, 10, and 100 detections per image, also with per-size variants.

\section{Method}
% ============================================================

We reproduce the Grounding DINO inference pipeline using the official \texttt{groundingdino-py} package~\cite{Liu23} without modifying the model architecture. Our work focuses on wrapping the official implementation with a modular Python interface for configurable inference, evaluation, and visualization.

\textbf{Scope and assumptions.} This reproduction makes several explicit simplifying assumptions: (1) \textbf{Zero-shot only}: we use the released pre-trained checkpoint without any fine-tuning on COCO training data. This means our results represent out-of-the-box performance rather than the best achievable with domain adaptation. (2) \textbf{Fixed thresholds}: detection uses hard confidence thresholds (box\_threshold, text\_threshold) rather than the full score-ranked evaluation used in official COCO benchmarks. As discussed in Section~4.4.1, this understates AP relative to threshold-free evaluation. (3) \textbf{Canonical category names}: our prompt uses only the standard 80 COCO category names without additional descriptions, synonyms, or context. Richer prompts may improve phrase-to-category matching. (4) \textbf{Single-scale inference}: we use the default single-scale inference (800px short side) without test-time augmentation or multi-scale testing. (5) \textbf{Single-dataset evaluation}: we evaluate exclusively on COCO val2017 without cross-dataset validation. These assumptions are explicitly documented to define the scope of our reproduction and distinguish it from the full evaluation protocol used in the original paper.

\subsection{Inference Pipeline}

The inference pipeline consists of four stages implemented across our modular Python codebase. The data flow through the system proceeds from image and text inputs through preprocessing, model inference, and postprocessing to produce COCO-format detection outputs.

\subsubsection{Stage 1: Model Loading}

The \texttt{GroundingDINOModel} class (\texttt{src/models/grounding\_dino.py}, 182 lines) wraps the official \texttt{groundingdino-py} \texttt{load\_model()} function. It validates the existence of the configuration file and checkpoint, loads the model weights onto the detected device (CUDA or CPU), and exposes a \texttt{predict()} method decorated with \texttt{@torch.no\_grad()} for memory-efficient inference. The configuration file (\texttt{GroundingDINO\_SwinT\_OGC.py}) specifies the exact architecture parameters and is loaded from the \texttt{groundingdino-py} package installation. The checkpoint file (662 MB) contains pre-trained weights for the Swin-T backbone, BERT encoder, deformable fusion layers, and detection head. Model loading includes explicit device detection: CUDA is preferred when available; otherwise the model falls back to CPU with a warning.

\subsubsection{Stage 2: Image Preprocessing}

Input images undergo a three-step transformation pipeline. First, OpenCV BGR images are converted to RGB format (PIL convention) via \texttt{cv2.cvtColor(image, cv2.COLOR\_BGR2RGB)}. Second, the image is resized using \texttt{RandomResize([800], max\_size=1333)}, which scales the shorter side to 800 pixels while constraining the longer side to at most 1,333 pixels, preserving the original aspect ratio. Third, the resized image is normalized using ImageNet statistics (mean $=[0.485, 0.456, 0.406]$, std $=[0.229, 0.224, 0.225]$) and converted from HWC to CHW tensor format. This preprocessing matches the training distribution of the Swin-T backbone, which was pre-trained on ImageNet-1K before detection-specific training.

\subsubsection{Stage 3: Text Encoding and Cross-Modal Fusion}

The text prompt undergoes tokenization by BERT's WordPiece tokenizer with a vocabulary of 30,522 tokens. The tokenizer splits words into subword units (e.g., ``toothbrush'' $\to$ [``tooth'', ``\#\#brush'']) and produces input IDs and attention masks of shape $(1, L)$ where $L \leq 256$ tokens. The tokenized text and the preprocessed image tensor of shape $(1, 3, H_{\text{resized}}, W_{\text{resized}})$ are processed jointly through the model. The Swin-T backbone extracts multi-scale visual features at strides $\{4, 8, 16, 32\}$. BERT produces contextualized token embeddings of shape $(1, L, 768)$. The six deformable attention fusion layers then enable each of the 900 object queries to attend to relevant spatial locations in the visual feature maps and relevant token positions in the text sequence. Each query outputs a bounding box prediction $(cx, cy, w, h)$ normalized to $[0,1]$, a confidence score $s \in [0,1]$, and a phrase index referencing the detected text span.

\subsubsection{Stage 4: Postprocessing}

Normalized box coordinates are converted from $(cx, cy, w, h)$ to pixel-space $(x_1, y_1, x_2, y_2)$ via \texttt{box\_cxcywh\_to\_xyxy()} in \texttt{src/utils/box\_ops.py}. For COCO evaluation, boxes are further converted to COCO format $(x, y, w, h)$. Bounding boxes are clamped to image boundaries to ensure valid coordinates, and detections with scores below the box\_threshold or text\_threshold are filtered. Degenerate boxes (width $\leq$ 0 or height $\leq$ 0, which can occur for very low-confidence detections) are discarded during the format conversion step.

\subsection{Prompt Design and Phrase-to-Category Mapping}

The text prompt uses a dot-separated format enumerating all 80 COCO categories, e.g., ``person . bicycle . car . motorcycle . airplane . bus . \ldots . toothbrush .''. The dot-separated format is the officially recommended approach~\cite{Liu23}: the BERT WordPiece tokenizer treats the period and surrounding spaces as independent tokens, creating clear boundaries between adjacent category names. This prevents the tokenizer from merging category names into ambiguous subword units — a failure mode that occurs with comma-separated or sentence-style formats.

After the model outputs detected phrases, a three-layer matching strategy in \texttt{src/datasets/coco\_categories.py} maps each phrase to a COCO category ID:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Exact match}: If the detected phrase exactly matches a COCO category name (case-insensitive), return the corresponding category ID.
    \item \textbf{Substring match}: If the phrase is a substring of a category name, or vice versa, return the longest matching category's ID. This handles partial detections (e.g., ``dining'' $\to$ ``dining table'').
    \item \textbf{Alias match}: A manually curated dictionary of 21 aliases resolves common variations (e.g., ``tv''/``television'' $\to$ ``tv'', ``sofa'' $\to$ ``couch'', ``person'' $\to$ ``person'', ``dining table'' $\to$ ``dining table'').
\end{enumerate}

Phrases that fail all three matching layers are discarded. These unmapped phrases (typically $\sim$2--5\% of detections, varying by threshold) contribute to reduced effective recall and are one cause of zero-detection images.

All configuration is managed through OmegaConf YAML files (\texttt{configs/grounding\_dino.yaml}), enabling CLI overrides for paths, thresholds, and hyperparameters. The evaluator implements the full COCO evaluation protocol as a seven-step pipeline: (1) load model and configuration, (2) load COCO annotations and category definitions, (3) iterate over image IDs running model inference, (4) map detected phrases to COCO category IDs, (5) convert bounding boxes from xyxy to COCO xywh format, (6) construct pycocotools COCO result objects, and (7) run COCOeval to compute all 12 AP and AR metrics. Results are exported in multiple formats: \texttt{predictions.json} (COCO-format detection results), \texttt{metrics.json} (full 12-metric summary), \texttt{eval.log} (detailed per-image statistics), and \texttt{phrase\_mapping.json} (detected-phrase-to-category mapping for debugging).

\subsection{Evaluation Protocol Details}

The COCO evaluation follows the official protocol implemented by pycocotools COCOeval. For each of the 80 COCO categories, a precision-recall curve is computed by varying the confidence score threshold across all detections of that category. Precision and recall are computed at each of 101 recall levels (0.00 to 1.00 in steps of 0.01), and AP is the area under the precision-recall curve. This process is repeated at each of 10 IoU thresholds (0.50 to 0.95, step 0.05), and the final AP is the average across all IoU thresholds and all categories. The per-size metrics (APS, APM, APL) are computed identically but restricted to ground-truth objects within each size range. Similarly, AR@k is computed as the maximum recall achieved with at most $k$ detections per image, averaged across IoU thresholds, categories, and size ranges.

Our evaluation pipeline includes two important implementation details. First, \textbf{degenerate box filtering}: boxes with width $\leq$ 0 or height $\leq$ 0 (which can arise from numerical precision issues during coordinate conversion) are detected and discarded before COCOeval to prevent invalid submissions. Second, \textbf{score-based ranking}: within each image, detections are sorted by confidence score before submission to COCOeval, ensuring that the top-$k$ detections used for AR@k computation are the highest-confidence ones. Both of these details follow the standard COCO evaluation protocol and are verified against the official COCO API documentation.

\subsection{Datasets}

We evaluate on the COCO 2017 validation set~\cite{Lin14}, which contains 5,000 images spanning 80 object categories with 36,781 ground-truth bounding box annotations (averaging $\sim$7.4 annotations per image). The annotation file \texttt{instances\_val2017.json} provides category definitions in COCO's $[x, y, \text{width}, \text{height}]$ format with non-consecutive category IDs (1--90 with gaps). The dataset exhibits a pronounced long-tail category distribution: frequent classes such as ``person'' dominate the instance count, while rare classes such as ``hair drier'' have fewer than 100 validation instances. All images are RGB color images of varying resolutions (from approximately $480 \times 640$ to full HD) drawn from diverse real-world scenes including indoor environments, outdoor street views, and crowded public spaces. For ablation experiments, we use a fixed subset of 500 images selected with random seed 42 for reproducibility.

\textbf{Inference speed.} Full COCO val2017 evaluation (5,000 images) completes in approximately 68 minutes on the RTX 4060, averaging 0.81 seconds per image. This includes model inference, postprocessing, and I/O overhead. Single-image inference exhibits a cold-start latency of approximately 6 seconds for the first image (CUDA kernel compilation and model warm-up), dropping to approximately 1.0 second for subsequent images. GPU memory usage during inference peaks at approximately 3.2 GB of the available 8 GB VRAM, leaving headroom for larger batch sizes or higher-resolution inputs.

The project is implemented as a config-driven Python package (\texttt{cv-project-2026}) with modular architecture. The implementation consists of: a model wrapper (\texttt{src/models/grounding\_dino.py}) wrapping the official \texttt{groundingdino-py} package; a predictor (\texttt{src/inference/predictor.py}) for single-image, batch, and directory-level inference; a COCO evaluator (\texttt{src/engine/evaluator.py}) integrating pycocotools COCOeval with format conversion; dataset utilities (\texttt{src/datasets/coco\_categories.py}) for category mapping and prompt construction; and a visualizer (\texttt{src/inference/visualizer.py}) for detection visualization with bounding box drawing. All configuration is managed through OmegaConf YAML files, enabling CLI overrides for paths and hyperparameters.

Default hyperparameters are box\_threshold=0.35 and text\_threshold=0.25, following the official Grounding DINO recommendations. Detection outputs are converted to COCO's submission format: $[\text{image\_id}, \text{category\_id}, [x, y, w, h], \text{score}]$.

\subsection{Metrics}

We use the standard COCO evaluation metrics computed by the official pycocotools COCOeval toolkit:

\begin{itemize}
    \item \textbf{AP}: Average Precision averaged over 10 IoU thresholds from 0.50 to 0.95 with step size 0.05. This is the primary ranking metric.
    \item \textbf{AP50}: AP at the single IoU threshold of 0.50, representing lenient localization requirements.
    \item \textbf{AP75}: AP at the single IoU threshold of 0.75, representing strict localization requirements.
    \item \textbf{APS, APM, APL}: AP broken down by object size — small (area $<32^2$ pixels), medium ($32^2 \leq \text{area} \leq 96^2$), and large (area $>96^2$).
    \item \textbf{AR@k}: Average Recall given $k$ detections per image ($k \in \{1, 10, 100\}$), with analogous per-size breakdowns (ARS, ARM, ARL).
\end{itemize}

All metrics are computed per-category and then averaged across the 80 COCO categories. The precision-recall curve for each category is computed by varying the confidence score threshold across all detections, with AP computed as the area under this curve at each of the 10 IoU thresholds.

In sentence-style format, additional function words (``There are'', ``and'', ``in the image'') introduce noise tokens that dilute the model's attention to category names.

\textbf{Practical implications.} These results confirm that the dot-separated format is not merely a convention but a functional requirement for Grounding DINO's text encoder. Any deployment using this model must format text prompts with dot separators between category names. The finding also suggests that Grounding DINO's BERT encoder was fine-tuned on dot-separated category lists during pre-training, creating a strong inductive bias toward this format.

\section{Discussion}

\subsection{Comparison with Reported Benchmarks}

The official Grounding DINO paper~\cite{Liu23} reports substantially higher zero-shot COCO performance for the Swin-T backbone than our result (AP $\approx$ 0.485 vs. our 0.4055). This discrepancy warrants careful analysis:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Evaluation protocol differences.} The official benchmark uses a GLIP-style evaluation protocol that tests only on previously unseen categories (approximately 50\% of COCO classes), whereas our evaluation uses the standard COCO protocol covering all 80 categories including common ones seen during pre-training. The GLIP protocol can yield higher AP by excluding ``easy'' base categories where the model would otherwise accumulate high-precision detections.
    \item \textbf{Threshold settings.} Official evaluations may use no fixed box\_threshold during AP computation, allowing the full precision-recall curve over all scored detections. Our fixed-threshold inference (box\_th=0.35) prunes low-confidence detections before COCOeval, reducing the recall ceiling. Our ablation confirms that lowering box\_th to 0.25 raises AP by approximately 2.5 points on the subset.
    \item \textbf{Prompt engineering.} The official prompt may include more descriptive category text beyond simple category names (e.g., category descriptions or synonyms), which can improve phrase-to-category matching. Our prompt uses only the canonical COCO category names.
    \item \textbf{Implementation version.} The \texttt{groundingdino-py} package (v0.4.0) may differ slightly from the original training codebase in preprocessing, postprocessing, or model serialization, introducing small numerical discrepancies.
\end{enumerate}

Given these factors, we consider our result (AP=0.4055) a realistic baseline for zero-shot Grounding DINO inference under standard COCO evaluation protocol with fixed thresholds, directly relevant to practitioners deploying the model as-is.

\subsection{Zero-Shot vs. Fine-Tuned Performance}

The Grounding DINO paper reports that fine-tuning the Swin-T model on COCO train2017 achieves AP $\approx$ 0.506, an improvement of approximately 10 AP points over zero-shot performance. This gap quantifies the domain adaptation benefit: COCO's specific image distribution, category appearance statistics, and annotation style differ from the pre-training datasets (Objects365, OpenImages, GoldG).

Several factors contribute to the zero-shot/fine-tuned gap. First, \textbf{domain shift in visual appearance}: COCO images have characteristic photographic styles, lighting conditions, and object contexts that differ from Objects365 and OpenImages. Fine-tuning adapts the Swin-T backbone's feature extractors to COCO's specific visual statistics. Second, \textbf{category-level calibration}: pre-training datasets have different category definitions and annotation granularity. Objects365 has 365 categories with coarser annotations, while COCO has 80 categories with dense, high-quality annotations. Fine-tuning allows the model to learn COCO-specific category boundaries and confidence calibration. Third, \textbf{background discrimination}: COCO's background distribution (indoor scenes, street views, natural landscapes) differs from pre-training data. Fine-tuning improves the model's ability to suppress false positives on COCO-specific background patterns.

The $\sim$10 AP point gap also suggests that approximately 20\% of COCO objects require dataset-specific visual knowledge that cannot be transferred from general pre-training alone. This has practical implications: for applications targeting specific visual domains (e.g., medical imaging, industrial inspection), some degree of domain-specific fine-tuning is likely necessary even with strong zero-shot models. Fine-tuning experiments on COCO train2017, while outside our current scope, represent the most direct path to closing the performance gap with reported benchmarks.

\subsection{Limitations Analysis}

\textbf{Small object detection bottleneck.} The 2.1$\times$ gap between APS (0.2587) and APL (0.5510) is the dominant limitation. The Swin-T backbone's 32$\times$ downsampling means that objects smaller than $32 \times 32$ pixels occupy at most a few feature map locations, providing insufficient spatial information for precise localization. This is a fundamental resolution constraint: at stride 32, the effective receptive field for the deepest feature maps is $\sim$128 pixels, meaning that small objects not only have limited feature representation but also ambiguous spatial context. While the 900 object queries can theoretically attend to small objects, the feature representation at those locations lacks the resolution for confident simultaneous classification and localization.

The scale of this problem on COCO is substantial: approximately 40\% of COCO val2017 ground-truth instances fall into the ``small'' category (area $< 32^2$), yet they account for the majority of false negatives in our evaluation. Multi-scale feature fusion (FPN-style) and higher input resolution (e.g., 1,333px short side instead of 800px) are architectural remedies beyond the scope of zero-shot inference with the released checkpoint. However, our threshold ablation shows that lowering box\_threshold to 0.25 recovers some small-object detections (APS improves from 0.265 to 0.306 on the subset), suggesting that post-hoc threshold optimization can partially mitigate, though not solve, this limitation.

\textbf{Prompt format brittleness.} The 62\% AP degradation with non-dot-separated prompts reveals a strong dependency on BERT's tokenization behavior. This limits the model's usability in applications where natural language queries are expected. Users must explicitly format category lists with dot separators, which is unintuitive for non-expert users and complicates integration with downstream natural language interfaces.

\textbf{Computational cost.} Full COCO evaluation requires approximately 68 minutes on a consumer GPU (RTX 4060), corresponding to 0.81 seconds per image. While acceptable for offline evaluation, this throughput is insufficient for real-time applications (target: $>$30 FPS). Architectural optimizations such as model distillation~\cite{Cheng24} or TensorRT deployment would be required for production use.

\textbf{Zero-detection images.} The 75 images (1.5\%) with zero detections represent a hard failure mode. Manual inspection suggests these images typically contain either extremely small objects, unusual viewpoints, or heavy occlusion. Adaptive thresholding — lowering thresholds for images where no high-confidence detections are found — could recover some of these cases.

\subsection{Future Work Directions}

Our findings suggest several concrete directions for extending this work:

\begin{enumerate}
    \item \textbf{Fine-tuning on COCO train2017}: Fine-tune the Swin-T checkpoint for 12 epochs using the training configuration in \texttt{configs/grounding\_dino.yaml} (AdamW, lr=$10^{-4}$, batch\_size=2) to approach the reported AP $\approx$ 0.506.
    \item \textbf{Cross-dataset evaluation}: Evaluate zero-shot performance on Objects365 validation set, OpenImages v6, and LVIS to measure generalization beyond COCO's 80 categories and image distribution.
    \item \textbf{Per-category AP analysis}: Identify specific COCO categories where Grounding DINO underperforms, which could guide targeted prompt engineering or fine-tuning data selection.
    \item \textbf{Adaptive threshold strategies}: Implement per-category or per-image threshold adaptation to recover detections in the 1.5\% of images currently producing zero outputs.
    \item \textbf{Model comparison}: Evaluate GLIP~\cite{Li22}, OWL-ViT~\cite{Minderer22}, and YOLO-World~\cite{Cheng24} under identical zero-shot conditions on COCO val2017 for a controlled comparison.
    \item \textbf{Deployment optimization}: Investigate TensorRT conversion or ONNX export for the Swin-T backbone to reduce inference latency below 50ms per image.
\end{enumerate}

\subsection{Reproducibility Considerations}

Several aspects of this reproduction required careful attention and may inform future efforts:

\begin{itemize}
    \item \textbf{Package compatibility}: The \texttt{groundingdino-py} package (v0.4.0) requires specific versions of \texttt{transformers} and \texttt{timm}. Version mismatches can cause silent numerical differences in BERT tokenization or image preprocessing.
    \item \textbf{CUDA environment}: PyTorch 2.7.1 with CUDA 11.8 was used; newer CUDA versions may require PyTorch nightly builds. The model checkpoint is architecture-specific and cannot be loaded with incompatible PyTorch versions.
    \item \textbf{Checkpoint integrity}: The checkpoint file (662 MB) should be verified via MD5 hash after download to prevent corruption. Corrupted checkpoints can produce plausible but incorrect outputs rather than clear errors.
    \item \textbf{Deterministic evaluation}: We use fixed random seeds (seed=42) for subset selection. Full val2017 evaluation is deterministic since all 5,000 images are processed.
    \item \textbf{GitHub repository}: The complete reproduction code, configuration files, and documentation are available in a public repository for independent verification and extension.
\end{itemize}

\subsection{Interactive Demo Interface}

As a supplementary component of this project, we developed an interactive web-based demonstration interface using Gradio (Abid et al., 2019) for real-time Grounding DINO inference. The interface (located in \texttt{frontend/}) provides the following capabilities:

\begin{itemize}
    \item \textbf{Live image upload and detection}: Users can upload arbitrary images and receive Grounding DINO detections within seconds. The interface displays annotated images with bounding boxes, category labels, and confidence scores overlaid.
    \item \textbf{Custom text prompts}: Users can input arbitrary text prompts (not limited to COCO categories), enabling open-vocabulary detection of any object describable in natural language. Example prompts include ``golden retriever . flower pot . wooden chair .'' for custom detection tasks.
    \item \textbf{Adjustable detection parameters}: Interactive sliders allow real-time adjustment of box\_threshold and text\_threshold, enabling users to explore the precision-recall tradeoff interactively. Lower thresholds reveal more detections (including false positives), while higher thresholds show only high-confidence results.
    \item \textbf{Multi-image batch processing}: The interface supports uploading multiple images simultaneously for batch detection, with results displayed in a gallery format for side-by-side comparison.
    \item \textbf{Result export}: Annotated images and detection results (bounding boxes, scores, phrases) can be downloaded in JSON format for further analysis or integration with external tools.
\end{itemize}

The Gradio interface is built as a lightweight wrapper around the \texttt{Predictor} class from \texttt{src/inference/predictor.py}, demonstrating the modularity and reusability of the project's software architecture. The same \texttt{Predictor} instance used for COCO evaluation serves inference requests from the web interface with no code duplication. The implementation requires only approximately 150 additional lines of code beyond the core inference library, highlighting how the modular architecture enables rapid prototyping of supplementary tools without modifying the underlying detection pipeline.

\subsection{Practical Deployment Considerations}

Beyond the experimental findings, several practical considerations emerged from this reproduction effort that are relevant to practitioners seeking to deploy Grounding DINO in real-world applications:

\textbf{Memory and throughput tradeoffs.} The Swin-T checkpoint requires approximately 2.5 GB of GPU memory for the model weights plus activations during inference with batch size 1. With 8 GB VRAM (RTX 4060), this leaves approximately 5.5 GB of headroom, which could support batch sizes up to 4 for improved throughput. However, the official \texttt{groundingdino-py} implementation does not natively support batched inference with variable image sizes, requiring either fixed-size inputs or sequential processing. For deployment scenarios requiring high throughput, batched inference with padding to uniform dimensions would be a valuable engineering improvement.

\textbf{Prompt engineering for custom vocabularies.} While our experiments focus on the fixed 80-class COCO vocabulary, Grounding DINO is designed for arbitrary text-described categories. Our preliminary tests with custom prompts (e.g., ``golden retriever . gaming chair . succulents .'') confirm that the model can detect categories entirely absent from COCO. However, performance degrades when category descriptions become multi-word phrases (e.g., ``traffic light'' already exists as a COCO class, but ``flashing yellow traffic light'' may confuse the phrase-to-box association). We recommend keeping category descriptions concise (1--2 words) and using the dot-separated format for custom vocabularies.

\textbf{Error recovery in production pipelines.} The 1.5\% zero-detection rate and occasional CUDA memory fragmentation during long-running evaluations suggest that production pipelines should include: (1) a fallback strategy for zero-detection images (e.g., lowering thresholds and re-inferring), (2) periodic GPU memory reset (every $\sim$500 images) to prevent fragmentation-related slowdowns, and (3) health checks on output distributions (e.g., alerting if the detection count per image drops below a configured minimum). Our evaluation pipeline already includes per-image error handling via try/except blocks, which proved essential during the 68-minute full val2017 run.

    \end{tabular}
\end{table}

Our core findings are threefold. First, Grounding DINO achieves $\text{AP}=0.4055$ ($\text{AP}_{50}=0.5317$) on full COCO val2017 in a zero-shot setting, demonstrating that open-vocabulary detection is viable without task-specific training. Second, threshold selection has substantial impact: AP ranges from 0.39 to 0.46 across three threshold settings, with box\_threshold=0.25 yielding the best results on our subset experiments. Third, the dot-separated prompt format is essential for the model's text encoder — alternative formats cause approximately 62\% AP degradation, a finding with direct implications for any practical deployment.

The primary limitation remains small object detection ($\text{AP}_{\text{S}}=0.2587$ vs. $\text{AP}_{\text{L}}=0.5510$), attributable to the Swin-T backbone's resolution constraints at 32$\times$ downsampling. The localization gap between AP50 and AP75 (9 percentage points) further suggests that bounding box regression, rather than category recognition, is the weaker component of the current zero-shot pipeline. These limitations, combined with the computational cost (68 minutes for full evaluation) and prompt format brittleness, define the boundary of practical zero-shot deployment today.

Future work should prioritize fine-tuning on COCO training data to approach the reported AP $\approx$ 0.506, cross-dataset evaluation to quantify generalization, and architectural or postprocessing improvements targeting small-object detection and localization precision. The modular, config-driven pipeline developed in this project provides a solid foundation for these extensions.
